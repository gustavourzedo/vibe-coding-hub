Playbook de atualizacao do hub

Esta pagina existe para um agente ou operador humano conseguir revisar o hub inteiro com custo baixo, duas vezes por dia, sem refazer trabalho desnecessario e sem misturar rumor com mudanca oficial.

Frequencia recomendada
2x por dia para pricing; 1x por dia para benchmarks
Metodo
Rules-first, diff-first, LLM only when needed
Objetivo
Manter o hub publico atualizado sem que o custo de monitoramento exploda

Procedimento-padrao do agente

1. Coleta

Baixar as paginas oficiais listadas abaixo, extrair texto limpo, normalizar e salvar hash + timestamp.

2. Deteccao

Se o hash mudou, comparar so os blocos relevantes: preco, quota, modelos, limites, claims, changelog e docs.

3. Julgamento

Usar LLM local apenas para resumir diferencas materiais e propor update da pagina afetada.

Regra de ouro: se nao houve mudanca textual relevante nas fontes oficiais, o agente nao chama LLM nenhum. Isso e o que deixa a rotina economica.

Regra de acao por tipo de mudanca

Tipo de mudanca Acao automatica Acao do agente
So mudou layout/CSS Ignorar Nao chamar LLM, nao abrir issue, nao editar o hub.
Mudou preco, quota ou overage Alta prioridade Gerar resumo, marcar pagina afetada, sugerir edicao e recalculo do ranking se necessario.
Mudou lista de modelos suportados Alta prioridade Atualizar benchmark/model watch e revisar aderencia tecnica.
Mudou benchmark oficial Media prioridade Atualizar pagina de benchmark e revisar narrativas tecnicas.
Mudou docs sem impacto material Baixa prioridade Registrar diff, mas adiar edicao publica ate acumular valor real.

Rotina economica 2x por dia

Rodada 1 - 08:00

  • Buscar paginas de pricing e docs mais sensiveis.
  • Comparar hashes e blocos extraidos.
  • Se nao houver diff material, encerrar em poucos segundos.

Rodada 2 - 20:00

  • Repetir pricing/docs.
  • Checar benchmark pages e posts oficiais mais recentes.
  • Executar sumarizacao local so para os itens que mudaram.
Forma economica certa: scraper e diff primeiro, LLM depois. O LLM local entra so para responder “essa mudanca e material?” e “qual pagina do hub precisa ser revista?”.

Arquitetura recomendada para automatizar

Camada 1

Coleta barata

Use `Playwright` ou `requests + readability` para extrair texto. Guarde hash, snapshot e trecho relevante em `SQLite`.

Camada 2

Decisao local

Rode um `14B-32B instruct local` via `Ollama` ou `vLLM` para classificar o diff como `cosmetico`, `material` ou `critico`.

Camada 3

Acao

So quando o diff for material: abrir issue, preparar patch HTML e opcionalmente sugerir commit.

Minha recomendacao para os seus dois servidores medios: `Servidor A` roda coleta, hashing, SQLite e scheduler. `Servidor B` roda o LLM local e o classificador de diff. O custo operacional cai muito porque o LLM nao fica ligado o tempo todo para navegar a web.

Stack sugerida

  • Scheduler: `cron` ou `n8n`.
  • Crawler: `Playwright` para sites JS-heavy e `requests` para docs simples.
  • Storage: `SQLite` com tabelas `sources`, `snapshots`, `diffs`, `actions`.
  • LLM local: um modelo instruct de `14B` a `32B`, suficiente para classificar diffs e resumir mudancas.
  • Fallback cloud: so para casos ambíguos ou quando o site quebra parse repetidamente.

Se a meta for economizar de verdade, eu evitaria um agente web-first rodando o dia todo. O melhor ROI aqui e `rules-first + diff-first + local summarizer`.
